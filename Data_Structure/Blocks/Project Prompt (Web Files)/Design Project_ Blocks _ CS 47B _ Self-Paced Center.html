Design Project: Blocks Project

Goals

This project has a number of interesting features. The problem to be solved is relatively complex. A solution will involve several time/space tradeoffs, and a solution program will be relatively large compared to other programs you have written for CS 47B. It also provides an opportunity to store something other than strings in a hash table.

Background

Those of you who spend much time in toy stores may be familiar with "sliding-block" puzzles. They consist of a number of rectangular blocks in a tray; the problem is to slide the pieces without lifting any out of the tray, until achieving a certain configuration. An example (from Winning Ways, E.R. Berlekamp et al., Academic Press, 1982) is shown in the diagram on the right.


This project is to write a program that generates a solution to such a puzzle. The first part is a design, the second an implementation of the design, and the third is an optimization and writeup.

Program interface

Your program will be named Solver.java. Its input will come from the command line and from files named there:

  • An optional first argument will be a string whose first two characters are "-o" and whose remaining characters specify information about what debugging output the program should produce. (You may choose the format of this information.) The string "-ooptions" should cause the program to print all the debugging options and the effect of each option. If the "-o" argument is not provided, your program should produce no output beyond that required to display a solution to the puzzle.

  • The next argument will name a file that specifies an initial tray configuration. Line 1 of this file will contain two positive integers, the length (number of rows) and width (number of columns) of the tray. Each subsequent line of this file will contain four nonnegative integers describing a block in the tray: the length and width of the block (both greater than 0), and the row and column of the upper left corner of the block. (The upper left corner of the tray is row 0, column 0.) Blocks are indistinguishable except for their size, and may appear in any order in this file. Thus the block configuration in figure 1 (reprinted below with row and column headers, and with blocks numbered to indicate their position in the example input) might be represented in the file as follows:

2 1 2 0

2 2 1 1
1 2 3 1
1 1 4 0
1 1 4 1
1 1 4 2
1 1 4 3


    Figure 1 — a sample initial configuration

  • The last argument will be the name of a file that specifies a desired final or goal configuration. This file is similar in format to the initial configuration file. Each line of this file contains four nonnegative integers: the length and width of the block (both greater than 0), and the desired position of the upper left corner of the block. This file will not necessarily contain entries for all blocks in the tray. Blocks may appear in any order in this file.

    The goal configuration mentioned in the diagram on the previous page is represented by the single line 

    	2 2 3 1

    If there were more than one 2-by-2 block in the initial configuration, the one-line goal would specify the position of any of the 2-by-2 blocks.

Again, if there are more than three 1-by-1 blocks in the initial configuration, it doesn't matter which three of them end up in the specified goal positions. Note also that the block positions may appear in any order in this file.

    goal configuration

    Figure 2 — a sample goal

Thus your program would be run with the UNIX command 

	java Solver.Solver [-oinfo] initialConfigFile goalConfigFile

where the -o argument is optional, info is the debugging information you supply, and initialConfigFile and goalConfigFile name the files containing the initial block configuration and the goal configuration respectively.

A solution to the puzzle will represent a sequence of position changes of blocks that, when starting with a specified initial configuration, ends up with blocks in the positions specified in the goal. The only legal moves are those that slide a block horizontally or verticallynot both at once—into adjacent empty space. Blocks may only be moved an integer number of spaces, and either the row or the column will be the same in the start position as in the end position for each move. (It's not legal to rotate the tray of blocks.)

Your program should produce a line of output for each block move that leads directly to a solution. Each such line will contain four integers: the starting row and column of the upper left corner of the moving block, followed by the upper left corner's updated coordinates. Example output appears below; the indicated moves, applied to the starting configuration of Figure 1, achieve the goal of Figure 2. (The annotations would not appear in the solution output.) 

	1 1 0 1		// move the 2x2 block up
	3 1 2 1		// move the 1x2 block up
	4 1 3 1		// move a 1x1 block up
	4 2 3 2		// move another 1x1 block up
	4 3 4 2		// move the rightmost 1x1 block over

If your program, run with debugging output disabled, finds a solution to the puzzle, it should exit normally after producing only output as just described, that is, the lines representing block moves that solve the puzzle. In particular, if the initial configuration satisfies the goal, your program should exit normally after producing no output. If your program cannot find a solution to the puzzle, it should exit with the call 

	System.exit(1);

again after producing no output.

You should check that command-line arguments are correctly specified, but you may assume that the initial and goal configuration files are free of errors. You may also assume that the length and width of a tray are no larger than 256.

Part 1 (4 points)

After finding a sponsor, you are to prepare a design for the Solver.Solver program. You need not produce the shortest possible sequence of moves, nor (for this part) will your program need to be exceptionally efficient.

The Solver.Solver program will need to search the tree of possible move sequences to find a solution to the puzzle. In your design, outline in detail how the program will do the following:

  • check if the goal configuration has been achieved;
  • generate moves possible in the current configuration, and choose the move to explore next;
  • make and unmake moves;
  • detect previously visited configurations in order to avoid infinite cycling.

Your sponsor will expect to see unambiguous descriptions of how the program will perform each of these operations.

Once you get your design approved, inform the Course Manager to get this approval recorded.

Part 2 (8 points)

Implement your design from part 1, make it work on small puzzles, and identify modifications to the program necessary to make it work efficiently with large or complicated puzzles.

Your program should conform to the interface requirements described at the start of this assignment:

  • Command line arguments specify debugging options (this argument is optional) and the initial and goal configuration files.

  • When no debugging option is specified, only one line per move, containing four integers, appears in the output. If the puzzle cannot be solved, the program should exit with the statement 

    	System.exit (1);
    Otherwise it should exit normally.

You should also associate debugging output with program events appropriately, and choose an appropriate debugging option for each set of output. Your debugging output facility should allow the user to select both the classes for which output is produced and the level of detail of output. Any interesting event that happens in your program-e.g. making/unmaking a move, encountering a previously seen configuration, determining the set of possible moves, or comparing a configuration with the goal-should be displayed by debugging output activated by some option. You should also incorporate output that will help you make implementation decisions about time/memory tradeoffs. Describe your debugging output facility in your writeup (see below). Your sponsor will ask you to defend your matching of output options with debugging output.

You are to include an "is OK" method in your class that represents the tray. When an appropriate debugging option is enabled, a call to this method should accompany each change to objects in the class. The "is OK" method should throw IllegalStateException with an informative message if it finds a problem, e.g. two blocks occupying the same space in the tray, or a block sitting outside the tray.

Program style and organization

It is likely in this assignment that you will need to revise code you implement in this version to be able to handle larger puzzles. Thus you should organize your program into classes that allow easy substitution of efficient code for inefficient code or of one algorithm for another (e.g. depth-first move processing for breadth-first) in each area.

As usual, your code should include informative comments for all your classes and methods. Use straightforward algorithms where possible. Give your variables and methods informative names. Declare all the data fields and methods within each class to have the proper public/private/protected/package qualifier. In particular, you are not allowed to make things public that would allow a user to corrupt your data structures (even if none of your own code would do this).

Plan for part 3

Producing a program that can solve large or complicated puzzles involves answering a number of questions.

  • Move generation will involve examining either the blocks in the tray or the empty space in the tray. Should the tray be stored as a list of blocks/empty spaces to optimize move generation, or should the locations in the tray be represented explicitly with a two-dimensional array? If the former, should blocks/spaces in the list be sorted?

  • What board representation optimizes the operation of checking whether the goal has been reached? If this representation is incompatible with implementations that optimize move generation, how should the conflict be resolved?

  • Should block moves of more than one space be considered? Why or why not?

  • Should the tree of possible move sequences be processed depth first, breadth first, or some other way?

  • The program needs to make and unmake moves. Again, a representation that optimizes these operations may not be so good for others. Determine how to evaluate tradeoffs among representations.

  • The program must detect configurations that have previously been seen in order to avoid infinite loops. Hashing is a good technique to apply here. What's a good hash function for configurations? The default limits for Java memory allocation may limit the maximum number of configurations that the table can contain. (The blocks in the puzzle diagrammed in Figure 1 may be moved into 65,880 different configurations.) How can this constraint be accommodated, and what effect does it have on other operations?

Some of these questions can be answered with careful analysis. Others require empirical evidence. Your program (with appropriate debugging options enabled) should produce enough information for you to make intelligent decisions about where its processing bottlenecks are.

Your sponsor will expect you to identify and defend, in your plan for part 3, at least three avenues for program improvement, along with design decisions that you feel you made correctly in the current version.

Again, once you get your program graded and an optimization plan approved, inform the Course Manager to get this approval recorded. If you spend too much time trying to make the program for this part efficient, you will lose pacing points and risk lowering your course grade.
